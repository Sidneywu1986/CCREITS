 公募 REITs 多智能体投资助手
 
 
 
 
 。

 一、在 OpenClaw 里更省 token 的几种做法
 
 
 1. 用多个子智能体，而不是一个超级智能体
 
 
 我更倾向于把系统拆成多个子智能体，每个智能体只负责一个相对清晰的环节。这么做有几个很直接的好处：
 
 
 
 
 每个智能体的职责更单一，提示词和工作空间更容易保持精简；
 
 
 
 
 
 
 
 各子智能体的上下文都是独立的，不会造成 token 的浪费；
 
 
 
 
 
 
 
 单轮对话上下文更短，不容易把无关历史全带进去；
 
 
 
 
 
 
 工具调用边界更清晰，哪个阶段该做什么不容易混；
 
 
 
 
 
 
 出问题时更容易定位；
 
 
 
 
 
 
 后续想替换某一块逻辑时，不需要重写整个系统。
 
 
 
 
 2. 让智能体之间互相调用
 
 
 OpenClaw 里，多智能体之间可以互相调用，即Agent To Agent。调用方式有两种：
 
 
 
 
 
 sessions_send
 
 
 
 ：更像把一条消息发给另一个已存在或目标明确的智能体，让它处理并回传；
 
 
 
 
 
 
 
 sessions_spawn
 
 
 
 ：更像临时创建一个隔离的子会话或子任务，让它独立跑完再把结果带回来。
 
 
 
 
 如果只是做“主控 → 固定子智能体”的日常协同，可以考虑
 
 
 
 sessions_send
 
 
 
 这种方式，因为它更接近“把任务发给某个明确角色去处理”。
 
 
 
 
 
 如果需要的是一个更临时、更隔离的子任务执行方式，可以考虑
 
 
 
 sessions_spawn
 
 
 
 。
 
 
 
 操作方式上，就是在
 
 
 
 openclaw.json
 
 
 
 里新增 agent，并给它们绑定对应的群或渠道，再开启会话工具和 agent-to-agent 能力。步骤很简单：
 
 
 第一步：创建智能体工作空间
 
 
 先在.openclaw/给你的智能体准备一个专属文件夹 workspace-
 
 
 
 &lt;agent-id&gt;
 
 
 
 ，然后创建必要的文件，包括：AGENTS.md、SOUL.md、USER.md等，这三个文件的作用：AGENTS.md 写智能体的操作指令和职责；SOUL.md 写智能体的人设、性格和边界；USER.md 可以选填用户信息，其他的TOOL和SKILLS等内容可以根据需要编写。
 
 
 第二步：在
 
 
 
 openclaw.json
 
 
 
 中新增智能体配置
 
 
 以
 
 
 
 sessions_send
 
 
 
 方式为例，下面是新增一个智能体并实现多智能体调用在
 
 
 
 openclaw.json
 
 
 
 中添加内容的示例：
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 {  "agents": {    "list": [      {        "id": "reits-analysis", // 智能体ID，后续调用它时会用到        "name": "REITs分析", // 展示名称        "workspace": "~/.openclaw/workspace-reits-analysis", // 这个智能体自己的工作空间目录        "model": {          "primary": "moonshot/kimi-k2.5" // 主模型        }      }    ]  },  "bindings": [    {      "agentId": "reits-analysis", // 把下面这个渠道绑定到哪个智能体      "match": {        "channel": "feishu", // 渠道类型，这里是飞书        "accountId": "default", // 使用哪个账号配置        "peer": {          "kind": "group", // 对端类型，这里是群          "id": "oc_bebd33***********" // 飞书群ID        }      }    }  ],  "tools": {    "sessions": {      "visibility": "all" // 允许会话工具可见，便于做多会话/多智能体协作    },    "agentToAgent": {      "enabled": true, // 开启智能体之间调用      "allow": ["main", "reits-analysis"] // 允许互相调用的智能体列表    }  }}
 
 
 当然，不想手动配置，上述步骤可以让 OpenClaw 里任意一个已有的agent帮你完成，只需和他发送消息“帮我创建一个新的智能体，配置信息和执行的任务是...”
 
 
 
 3. 不要把所有 LLM 过程都塞进 OpenClaw 里
 
 
 很多任务虽然最终要被 OpenClaw 调用，但
 
 
 
 不代表任务内部也一定要在 OpenClaw 的主会话里完成
 
 
 
 。就算需要LLM执行的任务，也可以从 OpenClaw 剥离出来，完全可以把一些复杂流程封装成工具，工具内部调用自己的 LLM、访问数据库、做多步处理，然后只把结果返回给 OpenClaw层面。
 
 
 这样做的好处非常明显：
 
 
 
 
 
 工具内部执行过程不会占用 OpenClaw 当前会话上下文；
 
 
 
 
 
 
 
 工具内部多轮处理不会把中间推理都塞进主会话；
 
 
 
 
 
 
 
 对智能体来说，它只看到“输入”和“结果”，上下文非常干净；
 
 
 
 
 
 
 
 响应速度通常会更快，因为主会话不用承载所有细节；
 
 
 
 
 
 
 
 在相同任务情况下，与 OpenClaw 内部的LLM执行相比，本地工具内部的LLM执行所消耗的token量真是小得太多了。
 
 
 
 
 
 就算不具备太强的代码开发能力，也可以考虑借助 Dify 这类低代码工作台，把某些流程做成 Workflow，再发布为 API，供 OpenClaw 调用。本质上，思路都是一样的：
 
 可以考虑
 
 
 
 
 
 把特定环节的、非调度性质的、多步的、重 token 的过程，尽量放到 OpenClaw 外部去做。
 
 
 
 
 我自己使用 OpenClaw 时，很多 tool 其实就是这么做的，例如：
 
 
 （1）
 
 多智能体REITs投资小助手
 
 
 
 后文要重点介绍的这个公募 REITs 投资小助手，表面上看是一个 OpenClaw 多智能体系统；但实际上，里面很多 tool 自身也都带有独立处理逻辑，甚至内部还会调用 LLM。只是这些过程都发生在 OpenClaw 主会话之外，后文重点介绍。
 
 
 （2）
 
 REITs新闻获取
 
 
 
 比如对于 REITs 新闻，重要来源之一是实时发布的公众号文章，我自己在本地会有程序爬特定公众号，并实时写入数据库。这样一来，“新闻查询”本身就可以变成一个很简单的 tool 或 skill：本质只是查数据库、做一点结果整理。然后再把这个tool 或 skill给到 OpenClaw 的agent。
 
 
 这样做的好处是：
 
 
 
 
 OpenClaw 里的agent不需要自己去网页里低效地找；
 
 
 
 
 
 
 新闻数据是我自己维护和沉淀的，质量更可控；
 
 
 
 
 
 
 后续无论是做问答、做总结、还是定时提醒，都很方便；
 
 
 
 
 
 
 甚至可以基于 OpenClaw 定时把新的新闻推给我，让我只关注真正值得看的变化。
 
 
 
 
 （3）
 
 REITs智能问答
 
 
 
 
 顺便提一句，
 
 
 去年我开发过一个 REITs AI 查询平台：
 
 
 
 
 
 www.reitschat.com.cn
 
 
 
 
 
 相关文章介绍在这里：
 
 
 公募REITs专属AI智能查询平台
 
 
 
 它是一个
 
 
 
 
 公募 REITs 专属 AI 智能查询平台
 
 
 
 
 
 ，基于多智能体技术，提供多板块公募 REITs 信息查询服务，
 
 
 项目后端源码也是开源的
 
 
 ：
 
 
 
 
 
 
 GitHub：https://github.com/adennng/reits-analysis-agent
 
 
 
 
 
 这个例子和我上面说的思路很一致：我完全可以把对该项目后端的调用能力封装成 tools或skills，再放到 OpenClaw 里，单独形成一个“查询智能体”或可以给任意一个智能体配备查询功能。这样的话，外部查询执行的大部分过程都不会占用 OpenClaw 会话 token，而且这个查询智能体也还能被别的智能体调用。
 
 
 总之：
 
 
 
 
 OpenClaw 不一定要亲自干完所有脏活累活，它更可以做一个统一的智能编排入口。
 
 
 
 
 二、OpenClaw 公募 REITs 投资小助手
 
 
 
 下面进入正文。
 
 
 
 （一）整体架构与运行流程
 
 
 我基于 OpenClaw 搭建了公募 REITs 多智能体投资小助手。它的核心目标，不是让单个大模型独立完成全部投资流程，而是将“日常投资执行链条”拆分为多个相对清晰、边界明确的环节，再由不同智能体分别负责
 
 
 
 复盘、分析、风控与策略代码生成
 
 
 
 ，最终由主控制智能体完成统一编排、状态推进与人与系统之间的交互。
 
 
 项目架构图如下：
 
 
 
 
 
 本系统的核心是一个
 
 
 
 
 按交易日滚动运行的半自动投资闭环
 
 
 
 
 。每个交易日收盘后，用户提供账户快照、成交明细和盘中数据文件，
 
 主控制智能体
 
 接收后创建本轮运行记录，并按顺序推进四个核心环节：
 
 
 
 
 
 复盘环节
 
 
 
 ：
 
 复盘智能体
 
 回顾上一轮及近期交易的预测准确性、交易合理性与脚本执行情况，输出结构化复盘信息入库，并定期增量更新为预测、交易、执行三份动态经验文档，以供分析环节参考。
 
 
 
 
 
 
 
 分析环节
 
 
 
 ：
 
 分析智能体
 
 先通过候选筛选工具缩小关注范围，再对候选基金逐一分析，形成初步交易建议草案。
 
 
 
 
 
 
 
 风控环节
 
 
 
 ：
 
 风控智能体
 
 以账户级风控规则审核并调整分析草案，输出最终交易方案。
 
 
 
 
 
 
 
 策略生成环节
 
 
 
 ：
 
 策略代码编写智能体
 
 将最终方案转化为可执行的 PTrade 策略脚本，并参考执行经验文档避免常见错误。
 
 
 
 
 脚本由用户检查后上传至 PTrade 交易平台，下一交易日自动执行。收盘后，真实执行结果再次输入系统，开启新一轮循环。同时，经验总结流程持续从复盘中抽取新数据，增量更新经验文档，反哺后续分析与策略生成，形成不断进化的学习回路。
 
 
 这一流程通过
 
 
 
 分阶段智能体分工、关键节点人工确认、工具承载重处理、经验闭环迭代
 
 
 
 ，将复杂的投资决策转化为可追踪、可优化、低 token 消耗的日度运行闭环。
 
 
 如果用一句话概括整个系统的运行主线，那就是：
 
 以某一个交易日为起点，围绕该日形成一轮 run，再让这轮 run 依次经过复盘、分析、风控、脚本生成四个阶段，最终输出下一交易日的可执行策略
 
 
 。
 
 
 
 
 （二）各智能体介绍
 
 
 1、主控制智能体：流程编排中枢
 
 
 主控制智能体是整个系统的入口与总控中心，承担以下职责：
 
 
 
 （1）接收每日输入并创建新一轮运行
 
 
 
 每个交易日收盘后，用户会把账户快照、成交明细和盘中交易窗口数据文件放入指定位置（或者通过飞书发送给该智能体），并通知该智能体开始工作。
 
 
 
 
 
 主控制智能体收到指令后，会先
 
 
 
 调用起始工具
 
 
 
 完成输入解析、目录初始化、在数据库中创建新记录等动作，从而为这一天建立一轮新的运行上下文。
 
 
 
 （2）推动流程按阶段向前
 
 
 
 主控制智能体不是一口气自动跑完整条链，而是按业务流程
 
 “半自动推进”（为了保留用户在关键节点进行人工确认的空间）：
 
 
 
 
 
 先完成运行创建；
 
 
 
 
 
 
 再询问是否开始复盘；
 
 
 
 
 
 
 复盘结束后询问是否进入分析；
 
 
 
 
 
 
 分析结束后询问是否进入风控；
 
 
 
 
 
 
 风控结束后询问是否生成策略代码。
 
 
 
 
 2、复盘智能体：把历史结果转化为经验
 
 
 
 （1）每日复盘
 
 
 
 复盘智能体负责检查上一轮和以及最近多轮中，系统在预测、交易建议和策略执行层面到底表现如何，并把这些结论转化为结构化的复盘结果并入库。
 
 
 它配备了三个tools，分别是预测复盘工具、交易复盘工具、执行复盘工具，该智能体依次
 
 调用三个工具
 
 进行
 
 三方面的复盘
 
 ，以判断：
 
 
 
 
 最近几个交易日的预测判断准不准；———是对于【预测服务】的预测准确性的分析（【预测服务】会对从四个方面（价格动量、公告信息、事件、市场整体）的角度进行分析，对未来3个时间维度的价格走势进行预测，详见后文的说明）
 
 
 
 
 
 
 最近几个交易日的交易建议合不合理；
 
 
 
 
 
 
 上一个交易日最终脚本和执行过程有没有偏差或问题。
 
 
 
 
 长期看，复盘智能体承担的是
 
 
 
 经验反馈器
 
 
 
 的角色，它把系统每一轮运行的真实结果重新送回经验层，为后续分析、风控与策略优化提供依据。
 
 
 
 （2）长期经验总结
 
 
 
 除了上述每日复盘以外，还有持续更新的历史经验教训
 
 总结文档
 
 。
 
 
 
 
 
 它不是静态知识库，而是动态经验层
 
 
 
 
 
 这里的文档是基于历史复盘结果持续更新出来的总结报告。系统会定期提取全部历史或特定一段时间内的预测复盘数据、交易复盘数据、代码执行复盘数据，然后分别形成对应的经验总结文档。（如果上一次已经总结过，那么下一次重点会放在新增数据和最近变化上，而不是每次完全从头重写）
 
 
 
 
 经验总结文档关注什么
 
 
 
 
 
 （1）
 
 
 预测经验总结文档
 
 会更偏向回答（是针对【预测服务】）：
 
 
 
 
 某类方向预测最近的倾向是什么；
 
 
 
 
 
 
 历史准确率如何、最近有没有变化；
 
 
 
 
 
 
 哪些场景更容易预测、哪些更容易失真；
 
 
 
 
 
 
 某些判断最近是否出现明显漂移；
 
 
 
 
 
 
 不同时间窗口下，系统的方向判断是否存在稳定偏好
 
 
 
 
 
 ...
 
 
 
 
 
 （2）
 
 
 交易经验总结文档
 
 更关注“建议是否合理”：
 
 
 
 
 在预测存在偏差时，系统是否能够识别风险并修正交易行为；
 
 
 
 
 
 
 在预测正确时，交易建议是否有效把握了机会；
 
 
 
 
 
 
 是否存在系统性保守、系统性激进或者过度交易倾向；
 
 
 
 
 
 
 哪些情况下应该交易却没交易，哪些情况下不该交易却做了多余动作；
 
 
 
 
 
 ...
 
 
 
 
 （3）
 
 执行经验总结文档
 
 主要聚焦在脚本与实际执行层:
 
 
 
 
 开盘不同时间段的成交量特点；
 
 
 
 
 
 
 历史上哪些时间段更容易成交、哪些更容易滑点或成交不足；
 
 
 
 
 
 
 代码执行结果是否满足交易需求；
 
 
 
 
 
 
 多次重复出现的报错点和执行风险点；
 
 
 
 
 
 
 对策略代码编写的具体建议。
 
 
 
 
 
 ...
 
 
 
 
 
 
 
 经验沉淀层如何反哺系统
 
 
 
 
 
 预测经验总结文档与交易经验总结文档，会和最近几个交易日的交易复盘数据一起提供给分析智能体的分析工具，作为形成交易建议时的重要参考之一。
 
 
 
 
 
 而执行经验总结文档，则会和最近几日的策略复盘数据一起提供给策略智能体，成为脚本生成时的参考输入之一。
 
 
 
 
 
 甚至历史数据足够多时，可以从预测经验文档中获取关键信息修正/优化【预测服务】。
 
 
 3、分析智能体：从候选筛选到交易建议生成
 
 
 分析智能体负责在收盘后基于当前账户、持仓、候选基金池以及已有知识
 
 ，对下一交易日应该关注什么、买什么、卖什么、持有什么做出初步建议。
 
 
 
 分析智能体依次调用“候选筛选”和“候选分析”两个工具实现先筛选，再分析：
 
 
 
 
 第一阶段先从更大的基金范围中筛出值得研究的候选对象；
 
 
 
 
 
 
 第二阶段再对候选基金逐一分析，形成具体的交易建议。
 
 
 
 
 
 （1）候选筛选工具
 
 
 
 候选筛选工具承担的是“缩小问题空间”的职责。
 
 
 判断：
 
 
 
 
 哪些基金值得进入本轮视野；
 
 
 
 
 
 
 哪些属于重点关注对象；
 
 
 
 
 
 
 哪些可以直接排除。
 
 
 
 
 
 （2）候选分析工具
 
 
 
 在候选池确定之后，候选分析工具再进一步对每个基金形成操作建议。
 
 
 它输出的重点不只是方向，还包括大致的调仓幅度的建议，也会形成相应的理由摘要。
 
 
 因此，
 
 分析智能体的产出，本质上是一个“初版交易决策草案”
 
 。
 
 
 （3）经验沉淀如何进入分析环节
 
 
 分析智能体并不是只看“今天的数据”，它还会参考一部分从历史复盘中沉淀出来的经验内容，
 
 包括特定时间内容的复盘信息以及经验总结文档
 
 。
 
 
 （4）
 
 
 
 
 关键
 
 
 
 
 
 ：分析智能体如何进行分析并形成交易建议？——与既有研究框架的关系
 
 
 
 分析智能体其实整个项目的是重点，关键在于如何进行分析。具体来说：
 
 
 
 
 会有另一个多智能体项目（与OpenClaw无关，先称之为
 
 【预测服务】
 
 
 ）
 
 
 在每个交易日结束后对市场中的每个基金从四个方面（价格动量、公告信息、事件、市场整体）的角度进行分析，对未来3个时间维度的价格走势进行预测、并提供判断的报告、并计算出“强度”信号，这部分数据会每日写入数据库中
 
 ；
 
 
 
 
 
 
 然后当前这个分析智能体的候选筛选工具会基于
 
 “强度”信号
 
 、
 
 已持仓基金
 
 、
 
 最近交易基金
 
 等维度从全市场中筛选出值得考虑的基金作为候选范围；
 
 
 
 
 
 
 然后候选分析工具会对候选范围内的全部基金逐一进行分析，所考虑的信息就包含【预测服务】生成的
 
 预测信息
 
 、
 
 判断报告
 
 、
 
 “强度”信号
 
 + 该基金最近几个交易日的
 
 交易复盘数据
 
 +
 
 预测经验总结文档
 
 +
 
 交易经验总结
 
 ，综合上述全部的信息做出是否要对该基金进行交易、交易的方向什么、大致的强度建议是什么。
 
 
 
 
 
 而【预测服务】其实就是我之前文章中提到的reits投资框架的里的“预测智能体”：
 
 
 
 
 
 之前文章链接：
 
 回撤减半，夏普翻番：AI驱动的多智能体公募REITs投资框架实证
 
 
 
 
 
 
 
 之前文章对应的论文：
 
 
 
 https://arxiv.org/pdf/2602.00082
 
 
 
 
 
 
 
 之前文章的投资框架对应的 GitHub 开源地址：
 
 
 
 https://github.com/adennng/REITs-MAS-LLM
 
 
 
 
 
 
 实际上，当前这个 OpenClaw 版本的投资助手，就是在上述reits投资框架的基础上做了很多工程化改造，本质上就是在往更工程化、更可日常运行的方向推进。
 
 
 
 4、风控智能体：把“想做”变成“能做”
 
 
 风控智能体位于分析之后、策略生成之前。它的职责是对分析智能体给出的建议做现实约束下的审核与调整。
 
 
 
 
 
 由于分析智能体对每个基金的交易建议仅是基于该
 
 单个基金层
 
 面所做出的，所以需要风控智能体
 
 基于账户级
 
 别的严格的风控要求，对每个基金交易建议进行调整，以满足整体的风控要求。
 
 
 5、策略代码编写智能体：把决策转为可执行脚本
 
 
 策略代码编写智能体是整个系统中最贴近交易落地的一环。它的核心职责是把前面经过分析与风控得到的最终交易方案，
 
 转换成可以直接提交到 PTrade 的脚本，以供下一个交易日执行
 
 。策略智能体的关键不在于“会不会写 Python”，而在于它承担的是一种
 
 
 
 执行翻译
 
 
 
 职责：
 
 
 
 
 把组合级决策翻译成订单级动作；
 
 
 
 
 
 
 把风险审核后的最终结果转化为交易平台可理解的执行表达。
 
 
 
 
 
 执行经验如何进入策略环节
 
 
 
 ：策略智能体也不是只机械地把风控结果翻译成代码。它会参考最近的
 
 脚本编写方案
 
 
 +
 
 
 执行经验总结文档
 
 
 +
 
 
 PTrade代码编写指南中的内容
 
 ，进而形成更接近“能落地、少犯重复错误”的脚本。
 
 
 
 最终交付物
 
 
 
 ：策略代码编写智能体将编写好的
 
 PTrade 的脚本
 
 放在特定目录下，用户检查后可上传至 PTrade 平台等待下一个交易日进行自动交易。
 
 
 
 6. 工具体系：让智能体真正“做事”
 
 
 
 
 从项目工程角度看，这个系统最大的特点之一，是
 
 
 每个子智能体背后都有明确的工具支撑，
 
 
 而不是纯文本输出。
 
 
 
 
 
 这套工具体系带来几个非常现实的好处：
 
 
 
 
 
 
 降低大模型自由发挥带来的不确定性；
 
 
 
 
 
 
 
 
 让系统更可维护；
 
 
 
 
 
 
 
 
 让重 token 过程外移。
 
 
 
 
 
 
 由于很多 tool 本身都可以独立完成复杂处理，甚至内部再调用 LLM，这些过程并不会全部压在 OpenClaw 当前会话里。这其实也是前面“省 token”经验在项目里的直接体现。
 
 
 
 （三）飞书群交互设计：让每个智能体既能独立又能协同
 
 
 
 每个智能体都拥有独立的飞书群，这使得系统在交互层面
 
 
 具备两种模式
 
 
 ：
 
 
 
 
 1. 集中式交互
 
 
 
 
 用户可以和主控制智能体沟通，让主控制智能体代为调用对应的子智能体进行工作。各子智能体会在自己独立的群里实时发送执行过程和结果，使用户清楚每个环节的执行情况，
 
 
 而子智能体发送的消息都不会占用主控制智能体的上下文，更“省 token”。
 
 
 
 
 2. 分布式交互
 
 
 
 
 用户也可以直接进入某个子智能体的群，与该智能体单独沟通。例如：
 
 
 
 
 
 
 只让复盘智能体单独跑一次历史复盘；
 
 
 
 
 
 
 
 
 只让分析智能体重新做一次候选分析；
 
 
 
 
 
 
 
 
 只让风控智能体检查某一轮是否需要调整；
 
 
 
 
 
 
 
 
 只让策略智能体重新生成一次脚本。
 
 
 
 
 
 （四）具体例子
 
 
 
 我在
 
 
 
 
 .openclaw/ 目录下创建了一个共享目录“
 
 
 
 
 reits-invest-project
 
 
 
 
 ”，里面会保存初始信息、经验总结文档、每轮执行时每个智能体执行的结果情况信息、最终生成的脚本等。
 
 
 
 
 1. 任务初始化
 
 
 用户在收盘后把账户快照、成交明细和盘中交易窗口数据文件放入指定位置，并在
 
 主控制智能体
 
 专属群里告诉该智能体，并让其开始工作。
 
 
 
 
 
 
 主控制智能体
 
 收到指令后，会
 
 解析文件信息、为本轮任务创建一个专属run_uuid、在共享目录下新建专属目录、在数据库中添加本次任务的信息等，
 
 
 
 完成本次任务的初始化。
 
 
 
 
 
 2. 复盘
 
 
 用户在复盘智能体专属群里告诉
 
 复盘智能体
 
 
 
 
 
 run_uuid，并要求其开始进行复盘
 
 
 
 
 。复盘智能体执行复盘并在群里实时发送任务执行进展以及最终执行结果。
 
 
 
 3. 分析
 
 
 用户在
 
 
 
 主控制智能体
 
 专属群
 
 
 告诉
 
 
 主控制智能体复盘已经完成，请调用分析智能体进行分析环节，
 
 
 
 主控制智能体
 
 收到后自主调用
 
 分析智能体
 
 向其传递
 
 
 
 
 run_uuid，并要求其开始工作
 
 
 
 
 
 。
 
 分析智能体
 
 
 在被调用后在其专属群里回复“收到”并开始进行分析，并
 
 
 
 
 在群里实时发送任务执行进展以及最终执行结果。
 
 
 
 
 
 
 4. 风控
 
 
 用户在
 
 
 
 主控制智能体
 
 专属群
 
 
 告诉
 
 
 主控制智能体需要进行风控环节，
 
 
 
 主控制智能体
 
 收到后自主调用
 
 风控智能体
 
 向其传递
 
 
 
 
 run_uuid，并要求其开始工作
 
 
 
 
 
 。
 
 风控智能体
 
 
 在被调用后在其专属群里回复“收到”并开始执行任务，并
 
 
 
 
 在群里实时发送任务执行进展以及最终执行结果。
 
 
 
 
 
 
 5. 策略脚本生成
 
 
 用户在
 
 
 
 主控制智能体
 
 专属群
 
 
 告诉
 
 
 主控制智能体需要进行下一环节——策略脚本生成，
 
 
 
 主控制智能体
 
 收到后自主调用
 
 策略智能体
 
 向其传递
 
 
 
 
 run_uuid，并要求其开始工作
 
 
 
 
 
 。
 
 策略智能体
 
 
 在被调用后在其专属群里回复“收到”并开始执行任务，并
 
 
 
 
 在群里实时发送任务执行进展以及最终生成了脚本。
 
 
 
 
 
 
 6. 检查
 
 
 用户在共享目录下找到本次任务
 
 
 
 
 run_uuid 对应的目录，在其中找到生成的脚本并进行检查。
 
 
 
 
 
 
 
 
 （五）总结
 
 
 本项目是我在使用 OpenClaw 过程中的一次探索性尝试。OpenClaw 本身具备丰富的扩展能力和广泛的应用场景，能够衍生出许多有趣而实用的功能，值得进一步深入挖掘与探索~
